AI-First Design

HoxCore was designed from the ground up to work with LLMs. Every design decision — YAML over proprietary formats, declarative over executable, human-readable schemas — reflects the goal of making registry data maximally useful as AI context.

There are two distinct ways HoxCore works with AI:

  • Passive — Your YAML entity files are fed directly to LLMs as structured context. No special tooling required.
  • Active — Entities reference specific LLM models and knowledge bases that HoxCore can invoke on your behalf.

YAML is naturally LLM-friendly

YAML is structured, human-readable, and fits well within LLM context windows. A single HoxCore entity file gives an LLM all it needs to understand a project: its purpose, status, relationships, tools, and knowledge sources.

The models Field

Any entity can reference one or more LLM models. These define which AI systems are associated with this project and how to reach them:

models:
  - id: assistant
    provider: ollama
    model: llama3
    description: "General project assistant"

  - id: code-reviewer
    provider: openai
    model: gpt-4o
    description: "Code review and refactoring suggestions"

  - id: doc-writer
    provider: anthropic
    model: claude-opus-4-5
    description: "Documentation generation"
FieldRequiredDescription
idYesLocal identifier for this model reference
providerYesModel provider: ollama, openai, anthropic, mistral, etc.
modelYesModel name as used by the provider
descriptionNoHuman note on this model's purpose for the project
base_urlNoCustom API endpoint (for self-hosted or proxy)
contextNoSystem prompt or context template for this model

Knowledge Bases

Knowledge bases point to collections of documents, notes, or files that provide context for AI interactions with this entity:

knowledge_bases:
  - id: kb-project-docs
    path: ./docs/
    description: "Project documentation and specs"

  - id: kb-codebase
    path: /path/to/codebase/
    description: "Source code for context"

  - id: kb-papers
    path: ~/research/papers/
    description: "Reference literature"

  - id: kb-notes
    path: ./notes/
    description: "Meeting notes and decisions"
FieldRequiredDescription
idYesLocal identifier for this knowledge base
pathYesFilesystem path (relative to registry root or absolute)
descriptionNoHuman description of the contents
includeNoGlob patterns to include (default: all)
excludeNoGlob patterns to exclude

Ollama (Local Models)

Ollama is the recommended provider for local, private AI assistance. No data leaves your machine.

# Install Ollama
curl -fsSL https://ollama.ai/install.sh | sh

# Pull a model
ollama pull llama3
ollama pull mistral
ollama pull codellama
# In your entity
models:
  - id: local-assistant
    provider: ollama
    model: llama3

  - id: code-helper
    provider: ollama
    model: codellama
    description: "Code generation and review"
# Ask project context
hxc ai ask P-001 "Summarize this project's current status and blockers"
hxc ai ask P-001 --model code-helper "Suggest improvements to the API design"

OpenAI

export OPENAI_API_KEY="sk-..."

Using Registry as LLM Context

Even without any active AI commands, your HoxCore registry is immediately useful as LLM context. Feed entity YAML directly to any LLM for instant project awareness:

Export single entity for context

# Export raw YAML to clipboard or pipe to LLM
hxc show P-001 --raw | pbcopy

# Export as structured JSON for API use
hxc show P-001 --output json

Export project context bundle

# Generate a context document for an LLM
hxc export projects --format markdown --status active > context.md

# Feed to an LLM via CLI tool (e.g. llm, aichat, etc.)
hxc show P-001 --raw | llm "What are the critical risks in this project?"

Programmatic use

import subprocess
import json

# Load project as structured data
result = subprocess.run(
    ["hxc", "show", "P-001", "--output", "json"],
    capture_output=True, text=True
)
project = json.loads(result.stdout)

# Pass to your LLM of choice
context = f"""
Project: {project['title']}
Status: {project['status']}
Description: {project['description']}
Tags: {', '.join(project.get('tags', []))}
"""

AI CLI Commands

HoxCore provides hxc ai subcommands for active model interaction:

# Ask a question in the context of a project
hxc ai ask P-001 "What should I prioritize this week?"

# Specify a model ID from the entity's models list
hxc ai ask P-001 --model code-helper "Review this architecture decision"

# Summarize a project
hxc ai summarize P-001

# Generate a status report
hxc ai report P-001 --format markdown

# Ask across multiple projects
hxc ai ask --tags ai,active "What are the common blockers across these projects?"

Active AI commands — alpha feature

The hxc ai command group is in active development. Features may change between releases. The reference model (storing models and knowledge_bases in YAML) is stable.

Best Practices

Keep knowledge bases focused

Reference specific, relevant directories rather than your entire filesystem. An LLM context window is finite — focused knowledge bases produce better results than sprawling ones.

Name models by role, not by provider

Use descriptive IDs like code-reviewer or doc-writer rather than gpt4 or claude. This makes it easy to swap providers without updating entity files.

Use local models for sensitive data

For entities containing confidential business context, configure Ollama local models. This ensures your project data never leaves your machine.

Treat the entity YAML as a project brief

Write your description fields as if you're briefing a new team member — or a new AI assistant. The more context you provide, the more useful the LLM responses will be.

Consistent model IDs across templates

If you use templates, standardize model IDs so your AI workflows work uniformly across all entities. Define a baseline set in your team templates.
